Documentación-patrón: Carpeta de Técnica Terapéutica — Sistema Knowledge Base

1. Visión general de la arquitectura

El sistema de Knowledge Base (KB) organiza el conocimiento clínico de técnicas terapéuticas en carpetas independientes, una por técnica. La carpeta ac/ (Activación Conductual) es la implementación de referencia. Cada carpeta de técnica es autocontenida y se conecta al sistema mediante un registry central por side-effect de importación.


Principio clave: Cada técnica comparte exactamente la misma estructura de 13 áreas temáticas definidas en un enum KBArea. Esto garantiza homogeneidad total entre técnicas, permitiendo que la UI y el motor clínico trabajen con cualquier técnica sin conocer sus detalles internos.

2.estructura de carpetas


src/knowledge/<id_tecnica>/

├── <id_tecnica>.manifest.ts  ← Manifiesto: metadatos + lazy loaders

├── index.ts          ← Auto-registro en el registry central

└── data/

  ├── area_01_conocimiento.json

  ├── area_02_objetivos_clinicos.json

  ├── area_03_herramientas_evaluacion.json

  ├── area_05_recursos_materiales.json

  ├── area_06_tecnicas_especificas.json

Convenciones de nombrado:


Carpeta: ID corto en minúsculas (ej: ac, exposicion, ds, rc)

Manifest: <id>.manifest.ts

Index: siempre index.ts

JSONs: siempre area_NN_<nombre_snake_case>.json con numeración 01-13


3. Archivo index.ts — Auto-registro

Propósito: Registrar la técnica en el registry central al ser importada.


Patrón exacto:



// <id>/index.ts — Auto-registro de la técnica <NombreTécnica>import { registerTechnique } from '../registry';import { <ID>_MANIFEST } from './<id>.manifest';registerTechnique(<ID>_MANIFEST);export { <ID>_MANIFEST };


Reglas:


    La constante exportada sigue la convención <ID_MAYÚSCULAS>_MANIFEST
    El import de registerTechnique viene de '../registry'
    La llamada a registerTechnique() se ejecuta como side-effect al importar
    Se debe agregar la línea import './<id>'; en registry.ts para que el auto-registro ocurra


4. Archivo <id>.manifest.ts — Manifiesto de técnica

Propósito: Define los metadatos de la técnica y mapea las 13 áreas a imports dinámicos (lazy loading, un chunk Vite por área).

patron exacto

/**

 * <id>.manifest.ts — Manifest de <NombreTécnica>

 *

 * Cada import() genera un chunk Vite separado en producción.

 * En dev, Vite sirve el JSON directamente (instant).

 */


import { KBArea } from '../types/technique.types';

import type { TechniqueManifest } from '../types/technique.types';


export const <ID>_MANIFEST: TechniqueManifest = {

 id: '<id>',          // Debe coincidir con TechniqueId

 nombre: '<Nombre Completo>',

 descripcion: '<Descripción clínica en 1-2 oraciones>',

 version: '1.0.0',

 fuentes_principales: [

  // 3-6 fuentes bibliográficas principales (formato APA simplificado)

  'Autor(es) (Año). Título del trabajo',

 ],

 areas: {

  [KBArea.CONOCIMIENTO]:       () => import('./data/area_01_conocimiento.json'),

  [KBArea.OBJETIVOS_CLINICOS]:    () => import('./data/area_02_objetivos_clinicos.json'),

  [KBArea.HERRAMIENTAS_EVALUACION]: () => import('./data/area_03_herramientas_evaluacion.json'),

  [KBArea.EJERCICIOS_TAREAS]:    () => import('./data/area_04_ejercicios_tareas.json'),

  [KBArea.RECURSOS_MATERIALES]:   () => import('./data/area_05_recursos_materiales.json'),

  [KBArea.TECNICAS_ESPECIFICAS]:   () => import('./data/area_06_tecnicas_especificas.json'),

  [KBArea.ESTRUCTURA_SESIONES]:   () => import('./data/area_07_estructura_sesiones.json'),

  [KBArea.VALORES_OBJETIVOS]:    () => import('./data/area_08_valores_objetivos.json'),

  [KBArea.BARRERAS]:         () => import('./data/area_09_barreras.json'),

  [KBArea.HABILIDADES_TERAPEUTA]:  () => import('./data/area_10_habilidades_terapeuta.json'),

  [KBArea.AREAS_VITALES]:      () => import('./data/area_11_areas_vitales.json'),

  [KBArea.SINTOMAS_PROBLEMAS]:    () => import('./data/area_12_sintomas_problemas.json'),

  [KBArea.ACTIVIDADES_POR_PROBLEMA]: () => import('./data/area_13_actividades_por_problema.json'),

 },

};

Campos del manifest:

Campos del manifest:


Campo / Tipo  /Obligatorio/  Descripción

id  /TechniqueId / Sí  /ID corto. Debe estar declarado en el type union TechniqueId en technique.types.ts

nombre/  string / Sí / Nombre completo legible

descripcion/  string  /Sí  /Descripción clínica concisa

version / string/ Sí / Semver

fuentes_principales / string[] / Sí / 3-6 referencias bibliográficas clave

areas / Record<KBArea, () => Promise<{default: unknown}>>  /Sí  /Las 13 áreas, todas obligatorias, como lazy loaders






Prerrequisito: Antes de crear una técnica nueva, agregar su ID al type union en technique.types.ts:


export type TechniqueId = 'ac' | 'exposicion' | 'ds' | 'rc' | '<nuevo_id>';



5. Los 13 archivos JSON de datos — Esquemas detallados

Cada JSON sigue un esquema específico. A continuación, la especificación exacta de cada uno:


5.1. area_01_conocimiento.json — Fundamentos teóricos

Propósito: Fundamentos teóricos, principios clave y evidencia científica de la técnica.



{ "area_id": "area_01_conocimiento",      // Siempre este valor fijo "nombre": "Área de Conocimiento",       // Siempre este valor fijo "descripcion": "<texto descriptivo>", "fuentes": ["<ref1>", "<ref2>", ...],     // 4-8 fuentes específicas del área "fundamentos_teoricos": {  "definicion": "<párrafo: qué es la técnica, en qué se basa, cuál es su foco>",  "origenes_historicos": [   {    "autor": "<Autor (Año)>",    "aportacion": "<qué aportó este autor al desarrollo de la técnica>"   }   // 3-6 hitos históricos ordenados cronológicamente  ],  "modelo_explicativo": "<párrafo: cómo la técnica explica el problema que trata>",  "mecanismo_de_cambio": "<párrafo: cómo produce el cambio terapéutico>" }, "principios_clave": [  {   "principio": "<nombre del principio>",   "explicacion": "<explicación clínica detallada en 1-3 oraciones>"  }  // 4-6 principios fundamentales ], "evidencia_cientifica": {  "metaanalisis": ["<cita y hallazgo principal>", ...],  // 2-4 metaanálisis  "eficacia_comparativa": "<párrafo: cómo se compara con otros tratamientos>",  "poblaciones_estudiadas": ["<población1>", ...]     // 5-8 poblaciones }}

Nivel de rigor exigido:


La definicion debe ser técnica, precisa, mencionar el mecanismo principal

Los origenes_historicos deben incluir autor real, año real, aportación verificable

Los principios_clave deben nombrar conceptos técnicos de la técnica (no genéricos)

La evidencia_cientifica debe citar metaanálisis reales con hallazgos concretos

6. Convenciones transversales de calidad

Aspecto  Requisito

Fuentes  Cada JSON tiene su propio array fuentes con 3-6 referencias reales verificables

IDs de ítems  Prefijo de 2 letras del área + número secuencial: HE01, ET03, TE05, RM02

Lenguaje  Español técnico-clínico. Sin jerga innecesaria pero sin simplificación excesiva

Ejemplos  Siempre concretos, con datos numéricos, pacientes ficticios realistas

Anclaje temporal  Ejercicios, herramientas y técnicas deben indicar en qué sesión se usan

No duplicación  Si un instrumento es genérico (BDI-II, etc.), se referencia con referencia_shared en vez de duplicar

area_id  Siempre coincide exactamente con el valor del enum KBArea correspondiente

nombre  Siempre coincide con el label humano en KB_AREA_LABELS


7. Integración con el sistema — Checklist para nueva técnica

Agregar el ID al type union TechniqueId en technique.types.ts

Crear la carpeta src/knowledge/<id>/

Crear data/ con los 13 JSONs (todos obligatorios, todos siguiendo los esquemas de arriba)

Crear <id>.manifest.ts con metadatos y lazy loaders

Crear index.ts con auto-registro

Agregar import './<id>'; en registry.ts

Validar que el tipo de areas en el manifest cubre los 13 KBArea

8. Ejemplo de consistencia interna (cross-references)

La carpeta ac/ demuestra cómo los JSONs se referencian entre sí de forma coherente:


Área 07 (sesiones) nombra herramientas del Área 03 ("Aplicar BDI-II y BADS")

Área 07 programa ejercicios del Área 04 ("Iniciar monitoreo de actividad")

Área 09 (barreras) nombra técnicas del Área 06 ("Implementar Hora de Preocuparse", "EHS")

Área 13 (actividades) implementa la graduación teórica descrita en el Área 01 (principios)

Área 03 referencia al shared para instrumentos genéricos ("referencia_shared": "inventarios_generales")

Esta red de referencias cruzadas asegura que la KB funcione como un sistema integrado, no como piezas sueltas.
